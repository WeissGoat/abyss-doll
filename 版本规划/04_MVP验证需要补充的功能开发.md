# MVP 验证需要补充的功能开发

> 本文档用于承接 MVP 白盒试玩前的功能缺口整理。目标不是扩展正式版系统，而是补齐验证“深渊产出 -> 工坊养成 -> 战力验证”闭环时必须可感知、可操作、可回归的最小功能。

## 1. 核心判断

当前 MVP 的数据量已经足够，功能侧的主要问题是：部分关键配置已经写入 JSON，但运行时代码尚未完整消费这些配置，导致策划想验证的体验可能不会实际发生。

优先级应围绕以下三件事：

*   关键奖励必须稳定送达，避免闭环被随机掉落截断。
*   局外养成必须能明确改变局内战斗或续航。
*   2 层怪物必须产生新的压迫感，而不是只换一组 HP/伤害。

**2026-05-10 执行口径：**

当前 `5.2` 到 `5.6` 的基础闭环已经具备可运行形态：局内背包整理、战斗掉落手动拾取、小镇卖出、撤离/战败结算、战斗/安全区消耗品使用均已进入可测状态。后续功能开发不再优先扩张系统边界，而是补齐“奖励稳定送达 -> 工坊成长 -> 下一次下潜验证”的关键链路。

本阶段新增总控文档：`版本规划/06_MVP白盒试玩收口推进计划.md`。

## 2. P0：必须补齐

### 2.1 Boss 保底掉落机制

**问题：** `LootPool` 当前是怪物配置内的单次权重抽取。文档要求【一阶动力核心】为 1 层守门人必掉，但如果仍放在普通权重池中，玩家可能打赢 Boss 也拿不到升级底盘的关键材料。同时，未来宝箱、事件、任务也会产生奖励，不能继续让每个调用方各写一套掉落逻辑。

**开发目标：**

*   新增 `RewardSystem` 模块，统一负责奖励解析。
*   新增 `/Rewards` 奖励表，支持 `Guaranteed` 保底奖励、`WeightedPools` 权重奖励、`Nothing` 空掉落与 `RewardRef` 奖励组合。
*   怪物、节点、事件等调用方只配置 `RewardID`。
*   战斗胜利结算时，`CombatNode` 通过 `RewardSystem` 解析怪物奖励，先生成保底掉落，再额外执行随机掉落。
*   结算 UI 需要把保底掉落和随机掉落都加入同一套拾取面板，仍然要求玩家手动塞进背包。
*   旧 `LootPool` 只作为迁移期 fallback，后续不再扩展。

**验收标准：**

*   击败 `elite_scrap_guard` 后，每次都出现【一阶动力核心】。
*   玩家如果背包塞不下核心，仍需通过丢弃/整理来完成取舍。
*   普通怪、Boss、节点奖励都可以通过 `RewardID` 复用同一套解析流程。

### 2.2 出售规则保护材料与核心物

**问题：** 当前小镇卖出循环按 `BaseValue` 直接出售仓库物品，`SellAll` 容易把制造材料和 0G 核心物一起清掉。MVP 要验证“卖钱 vs 留材料”的决策，不能让一个按钮无差别销毁成长资源。

**开发目标：**

*   `SellItem` 和 `SellAllStashItems` 增加可售卖判定。
*   默认只允许出售纯卖钱战利品。
*   材料、核心物、任务物默认不可被全部卖出。
*   单件出售按钮对不可售物品应禁用或显示原因。

**建议规则：**

*   `ItemType == Loot` 且不含 `Material` / `CoreMaterial` / `Cursed` 标签：可出售。
*   `ItemType == QuestItem`：不可出售。
*   若后续需要“材料也可卖”，应显式加 `CanSell: true` 字段，而不是依赖 `BaseValue > 0`。

**验收标准：**

*   点击“全部卖出”不会移除【一阶动力核心】。
*   点击“全部卖出”不会移除用于制造义体的核心材料。
*   纯卖钱战利品能正常转化为金币。

### 2.3 义体配置生效链路

**问题：** 义体 JSON 使用 `PassiveEffect` 简写结构，而运行时代码主要读取 `PassiveEffects` 的 `EffectData` 列表。结果可能是制造并装备义体后，玩家看不到伤害或 SAN 续航变化。

**开发目标：**

*   统一义体配置结构，推荐全部改为 `PassiveEffects: EffectData[]`。
*   或在加载时兼容 `PassiveEffect` 并转成 `PassiveEffects`。
*   确认 `DamageMultiplier` 能按标签只增幅近战武器。
*   实现或接入 `RestoreSANOnCombatEnd` 效果。
*   工坊 UI 增加制造/装备义体入口，调用现有 `CraftAndEquipProsthetic` 链路。
*   制造后刷新金币、材料、已装备义体和战斗派生数值提示。

**验收标准：**

*   装备【动力臂增幅插件】后，近战武器面板或战斗日志中的实际伤害提高。
*   装备【稳压散热插件】后，每场战斗胜利结算时恢复 2 点 SAN。
*   两个义体都能通过工坊制造、扣材料、装备并参与下一场战斗。
*   玩家能在工坊界面看到义体制造入口，而不是只能通过测试代码调用。

## 3. P1：强烈建议补齐

### 3.1 怪物网格干涉效果

**问题：** 2 层怪物配置了 `GridInterference`，但怪物战斗逻辑暂未实际执行这些效果。这样 2 层体验会退化为“更高数值的小怪”，无法验证“怪物越来越恶心”。

**开发目标：**

*   `ReduceDamage`：酸液软体攻击或回合效果触发时，使玩家随机一把武器下回合伤害减半。
*   `AddCursedItem`：畸变融合体触发时，向玩家背包塞入 1x1 诅咒废件。
*   诅咒废件至少需要具备不可出售、不可普通丢弃或结算前占格的规则之一。

**验收标准：**

*   2 层战斗中玩家能明显感到武器输出被打断。
*   畸变融合体能制造背包空间压力，而不是只造成高额伤害。

### 3.2 Toxic 标签的携带代价

**问题：** 【污染滤芯】带 `Toxic` 标签，但当前没有看到运行时代码消费该标签。若没有 SAN 代价，它只是高价 2x2 战利品，不具备“高风险高回报”。

**开发目标：**

*   在移动到下一个深渊节点时，扫描背包内 `Toxic` 物品。
*   每件 Toxic 物品追加 SAN 消耗，MVP 建议每节点额外 -1 SAN。
*   UI 或结算日志标明毒性物品造成的额外消耗。

**验收标准：**

*   背包中携带【污染滤芯】时，移动节点的 SAN 消耗高于基础层数消耗。
*   玩家能在收益和续航之间做出取舍。

### 3.3 安全区的时间/恢复成本

**问题：** MVP 文档中安全区是“可消耗时间恢复 HP/SAN”的路线抉择点。当前若安全区只作为免费恢复点，会削弱撤离压力。

**开发目标：**

*   安全区至少提供一个明确成本：消耗 1 天、消耗金币、或只能使用背包内消耗品。
*   若时间系统未接入，MVP 可先用“本次下潜收益倍率降低”或“安全区只能使用消耗品”替代。

**验收标准：**

*   玩家不会无脑选择安全区。
*   安全区能成为“续航 vs 收益”的决策。

## 4. P2：可以暂缓，但建议记录

### 4.1 经济压力与账单系统

当前小镇阶段已经有卖出和升级回流，但还没有正式的房租/账单压力。MVP 白盒试玩可以先不接完整时间账单，但需要在后续验证中加入最小经济压力，否则玩家没有“必须下深渊”的长期约束。

### 4.2 二阶素材的后续用途

2 层掉落二阶素材可以先作为“下一阶段目标”存在，不必立刻接完整制造树。但至少应能在结算和仓库中被识别，避免 2 层胜利没有成长期待。

### 4.3 美术资源表现接线

当前 `VisualAssetRegistry` 已经能登记物品、怪物、节点、背景、人偶、底盘和义体素材，但运行时主要只有物品图标被消费。背景、怪物头像、地图节点图标属于表现接线工作，不阻断规则闭环，但会影响白盒试玩的可读性。

**开发目标：**

*   战斗敌人卡片读取 `PortraitID` 或默认 `monster_{MonsterID}_portrait`。
*   深渊地图节点读取 `node_combat_icon`、`node_safe_room_icon`、`node_boss_icon`。
*   工坊、深渊地图、战斗界面接入对应背景图。
*   缺失资源使用 `ui_missing_sprite`，不阻断流程。

---

## 5. 推荐开发顺序

1.  `RewardSystem`、`/Rewards` 奖励表与 Boss 保底掉落。
2.  出售规则保护材料与核心物。
3.  义体 `PassiveEffects` 生效链路与工坊 UI 制造入口。
4.  `RestoreSANOnCombatEnd`。
5.  正式 MVP 初始装与测试初始装拆分。
6.  怪物 `ReduceDamage` 与 `AddCursedItem`。
7.  `Toxic` 携带 SAN 代价。
8.  安全区成本。
9.  怪物头像、节点图标、背景图表现接线。

完成 1-5 后，可以开始第一轮 MVP 白盒试玩；完成 6-9 后，再进入第二轮体验调优。
