# MVP 客户端开发里程碑与节点规划

> **定位说明：** 本文档是客户端（Unity + C#）正式进入开发阶段的“施工图纸”。
> **开发原则：** 严格遵循“自底向上、数据先行、表现延后”的后端转客户端开发思维。每一个里程碑都必须能在不依赖花哨 UI 的情况下，通过控制台（Debug.Log）或自动化后端测试框架（AutoTestDaemon）跑通闭环。
> **必读文档:** `版本规划\01_MVP美术与UI需求清单`
> **当前收口计划:** `版本规划\06_MVP白盒试玩收口推进计划.md`
---

## 🏆 阶段一：数据地基与配置连通 (Data & Config) 
**目标：** 让 Unity 引擎能够正确认识我们定义的“万物实体”，并成功把策划写的 JSON 配置表全数读入内存。

*   **节点 1.1：编写纯 C# 数据容器**
    *   建立 `ItemEntity.cs` 及其核心组件（`ItemGridComponent`, `ItemCombatComponent`, `EffectData`）。
    *   建立 `DollEntity.cs` 及其组件（状态、三维、底盘 `ChassisComponent`）。
    *   建立 `PlayerProfile.cs`（全局资产与大仓库）。
*   **节点 1.2：引入 JSON 解析库**
    *   在 Unity 中引入 `Newtonsoft.Json`。
*   **节点 1.3：编写 `ConfigManager` (配置中心)**
    *   实现读取 `StreamingAssets/Configs` 目录下所有 JSON 文件的逻辑。

## 🏆 阶段二：核心算法与效果引擎 (Grid & Effect Framework)
**目标：** 实现全游戏最硬核的算法逻辑——“背包俄罗斯方块”与“连结拼图效果”。

*   **节点 2.1：实现 `BackpackGrid` 容器**
    *   完成二维数组 `string[,]` 的初始化（基于底盘的死格遮罩）。
    *   完成核心难点：形状旋转（矩阵变换）与边界/碰撞检测（`CanPlaceItem`）。
*   **节点 2.2：搭建 `EffectFactory` (效果工厂)**
    *   实现 `EffectBase` 基类，基于 C# 反射机制动态查找派生类。
    *   编写第一个实际效果：`DamageMultiplierEffect` (增伤光环)。
*   **节点 2.3：实现 `GridSolver` (网格解算器)**
    *   完成网格拓扑扫描：当物品放入包中时，能够准确找到其周围的物品，并应用 Effect。

## 🏆 阶段三：战斗状态机与阵营博弈 (Combat State Machine)
**目标：** 跑通基于 AP 和阵营的手动回合制战斗逻辑。

*   **节点 3.1：封装 `FighterEntity` 与 `CombatFaction`**
    *   实现 `DollFighter` 和 `MonsterFighter`，分离底层数据与战斗临时数据（`RuntimeHP`, `RuntimeShield`）。
    *   实现阵营的存活检测（`IsWipedOut`）与回合初/回合末钩子。
*   **节点 3.2：搭建 `CombatSystem` (战斗总控)**
    *   实现状态机：`PlayerTurn` -> `EnemyTurn` -> `End`。
    *   实现 AP 扣除逻辑与怪物 AI 的简易索敌攻击逻辑。
*   **节点 3.3：深渊节点树 `DungeonManager`**
    *   实现 `NodeBase`，跑通“扣除移动税 SAN -> 触发遭遇战 -> 胜利掉落”的流程。
*   **【新增】节点 3.4：自动化测试沙盒 (AutoTestDaemon)**
    *   **架构落地：** 引入 `AutoTestDaemon` 和 `BackendSmokeTest`，实现零 UI 挂载的纯后端“幽灵”闭环验证流。

## 🏆 阶段四：表现层解耦与事件总线 (View & EventBus) 
**目标：** 开始处理 Unity 的画面表现，用松耦合的方式把 UI 和底层的黑盒绑在一起。

*   **节点 4.1：实现 `EventBus` (全局事件中心)**
    *   定义核心事件：`OnItemPlacedEvent`, `OnHPChangedEvent`, `OnAPChangedEvent`, `OnTurnChangedEvent`。
*   **节点 4.2：实现 `GameRoot` 与 `CoreBackend` (唯一入口)**
    *   在场景中挂载 `GameRoot`，实例化 `CoreBackend` 黑盒，打通前后端桥梁。
*   **节点 4.3：UI 网格生成与拖拽 (Pure UGUI)**
    *   制作右侧的网格背景（使用 `GridGenerator` 动态生成）。
    *   实现 `DraggableItemUI` 和 `GridSlotUI` 的精准像素拖拽、偏移值计算与边缘越界阻挡。
*   **节点 4.4：常规 UI 数据绑定 (Data Binding) **
    *   实现血条、SAN条、结束回合按钮（基于纯 UGUI 开发，废弃 UI Toolkit），并在 `HUDController` 中完成事件监听。
*   **节点 4.5：表现队列 (Visual Queue) **
    *   拦截战斗结算的瞬间数据变化，排队执行动画（飞弹、飘字、受击闪白），实现流畅的战斗视觉反馈，不与底层毫秒级结算产生冲突。

## 🏆 阶段五：系统缝合与 MVP 闭环体验 (The Loop) 
**目标：** 把局外的工坊养成和局内的深渊战斗彻底缝合，形成闭环。

*   **节点 5.1：实现 `WorkshopSystem`**
    *   打通升级底盘（网格变大）和制造义体的扣除逻辑。
*   **节点 5.2：局内背包管理收口**
    *   在深渊地图界面增加【打开背包】入口。
    *   支持在非战斗状态下整理背包、重排格位，并主动将物品移出局内背包。
    *   确立 MVP 规则：从局内背包移出的物品视为现场丢弃，不自动返回仓库。
*   **节点 5.3：战斗掉落拾取界面**
    *   战斗胜利后不再直接把掉落写入背包。
    *   弹出掉落面板，展示本次战利品清单。
    *   玩家必须手动将战利品拖入局内背包，完成“拿什么/放弃什么/丢掉什么”的取舍。
*   **节点 5.4：小镇卖出循环**
    *   将撤离带回的战利品存入 `StashInventory`（大仓库）。
    *   在小镇界面增加最小可用的卖出能力，将战利品转化为金币。
    *   让金币能够明确回流到底盘升级与义体制造中。
*   **节点 5.5：战败/撤离结算收口**
    *   撤离成功时，只结算最终留在局内背包中的物品。
    *   战败时，清空局内背包内容，并保留局外仓库物资。
    *   结算界面需要清晰展示：本次拾取、本次带出、本次损失。
*   **节点 5.6：消耗品使用闭环**
    *   打通战斗中使用修复剂、镇静剂的逻辑。
    *   明确 AP 消耗、目标规则、使用反馈与数值刷新。
*   **节点 5.7：内容填充（MVP 最低差异化）**
    *   补齐纯卖钱战利品、制造素材、消耗品、过渡装备四类核心内容。
    *   目标不是堆数量，而是让玩家在空间、战力、经济之间做出真实取舍。
*   **节点 5.8：MVP 白盒试玩**
    *   **终极验收：** 完整体验一遍《01_数值模型沙盘推演方法论》中的流程。
    *   验收重点从“系统能跑”提升为“玩家是否真的被迫做背包取舍与经济决策”。

## 2026-05-10 状态复盘：进入白盒试玩收口

当前项目已完成数据地基、核心战斗、事件总线、基础 UI 骨架、深渊地图背包整理、战斗掉落手动拾取、小镇卖出、撤离/战败结算、战斗/安全区消耗品使用、视觉资源 Registry 与自动化测试闭环。

因此后续不再以“继续铺更多系统”为主，而是进入 **MVP 白盒试玩收口**：

*   **P0：闭环脊梁**：RewardSystem 与 Boss 保底掉落、出售规则保护、义体制造与生效链路。
*   **P1：深渊压力**：怪物网格干涉、Toxic 携带 SAN 代价、安全区成本。
*   **P2：表现接线**：怪物头像、地图节点图标、工坊/深渊/战斗背景图。

阶段五中 `5.2` 到 `5.6` 已经具备可运行形态，`5.7` 与 `5.8` 的重点应改为“验证已有内容是否承担清晰职责”，而不是继续增加大量新物品或新怪物。

详细执行顺序见：`版本规划/06_MVP白盒试玩收口推进计划.md`。
