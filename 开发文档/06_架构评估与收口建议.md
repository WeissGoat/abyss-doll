# 架构评估与收口建议

> **定位：** 记录当前客户端架构的优点、主要风险与后续收口方向，作为 Alpha 阶段继续推进时的架构备忘录。
> **更新时间：** 2026-04-26

---

## 1. 当前架构的主要优点

### 1.1 无头后端方向正确
项目从一开始就把核心业务逻辑放在纯 C# 黑盒中，Unity 更接近 View 与运行时适配层。  
这让战斗、深渊、工坊、背包等规则可以在无场景、无表现层的状态下直接测试。

### 1.2 数据驱动骨架已经成立
`StreamingAssets/Configs` + `ConfigManager` 的结构已经能够承接后续大量内容填充。  
当前项目的真正“源头”不是 Inspector，而是配置文件，这对数值迭代和 AI 协作都非常友好。

### 1.3 自动化回归是核心护城河
`AutoTestDaemon` + Headless `VisualQueue` 的组合已经能够在极低成本下完成回归。  
对于本项目这种高规则密度、强状态机、强配置联动的游戏，这是极高价值的基础设施。

### 1.4 场景骨架代码生成路线值得保留
`MVPEditorSetup` 让 UI 骨架和场景结构具备幂等重建能力。  
这避免了原型阶段最常见的“手工拖拽越来越脏”的问题。

---

## 2. 当前最值得警惕的结构风险

### 2.1 领域推进事件与表现排队事件容易缠在一起
如果“战斗结束”“地图前进”“结算完成”这类领域事件本身也进入 `VisualQueue`，逻辑推进就会被动画时序绑架。  
这会让测试、热重载、异常恢复和状态追踪都变得脆弱。

### 2.2 全局静态状态较多
`GameRoot.Core`、多个静态 `EventBus`、`VisualQueue` 当前仍可控，但随着内容规模增长，热重载与长时间编辑器会话中容易出现状态串线。

### 2.3 UI 切屏逻辑需要正式状态机化
如果继续以多个 `SetActive` 和分散入口推进，后面加入更多面板、弹窗、战斗结算分支时，状态切换会越来越难维护。

### 2.4 配置系统还缺一层“引用校验”
当前配置已经能加载，但仍缺 `ValidateAllConfigs()` 级别的引用一致性校验。  
随着怪物、掉落、配方、效果和地牢层数增加，配置引用断链会比代码错误更常见。

## 3. 已先行实施的两项收口

### 3.1 领域事件与表现事件分层
本轮调整后，`DungeonEventBus` 被收回为**同步领域事件总线**：

* `OnLayerLoaded`
* `OnNodeEntered`
* `OnSafeRoomEntered`
* `OnCombatNodeCleared`
* `OnDungeonEvacuated`
* `OnDungeonDefeated`
* `OnDungeonSettlementPrepared`
* `OnDungeonSettled`

这些事件现在默认立即触发，不再进入 `VisualQueue`。  
需要延时的撤离/战败/结算展示，改为由 `GameFlowController` 在表现层排队处理。

### 3.2 `GameFlowController` 正式状态机化
本轮将屏幕切换统一收口为：

* `Workshop`
* `DungeonMap`
* `Combat`
* `SafeRoom`
* `Settlement`

入口统一通过 `TransitionToScreen(...)` 处理，屏幕激活、日志、面板初始化和 payload 分发不再散落在多个地方。  
这为后续加入更多 UI 分支和中间态提供了稳定扩展点。

## 4. 推荐的后续架构动作

### 4.1 给配置系统补完整性校验
建议新增 `ConfigValidationReport` 与 `ValidateAllConfigs()`：

* 校验 `Doll -> Chassis`
* 校验 `Dungeon -> BossNode`
* 校验 `CraftingRecipe -> Prosthetic`
* 校验 `Item/Monster -> Effect`

校验结果直接落地到 `Logs/`，并允许测试框架单独触发。

### 4.2 显式引入 Session 生命周期
建议逐步让“本局运行态”拥有清晰的 `Init / Reset / Dispose` 边界，减少对静态全局状态的依赖。  
如果未来需要存档、继续运行、多局回放或更复杂的自动化，这一步会很重要。

### 4.3 拆分大型场景构建器
`MVPEditorSetup` 当前仍适合作为 MVP 工具，但 Alpha 阶段建议拆成多个 Builder：

* RootBuilder
* WorkshopScreenBuilder
* CombatScreenBuilder
* DungeonScreenBuilder
* SettlementScreenBuilder

### 4.4 让地牢生成可复现
建议后续为地图生成引入 `seed`。  
这样测试、数值回放、bug 复盘和玩家反馈复现都会更稳定。

---

## 5. 当前不建议做的事

* 不建议现在引入大型 DI 框架
* 不建议把 JSON 全量改成 ScriptableObject
* 不建议为了“现代化”而切换到 ECS / DOTS

当前项目最重要的优势是：

* 规则可验证
* 闭环可回归
* 配置可迭代
* 前后端边界清楚

继续沿这条线收口，比更换技术名词更有价值。
