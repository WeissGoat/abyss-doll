# 视觉资源系统程序开发规范

> **定位：** 规定程序侧如何把 JSON 配置、Unity 资源、Prefab 和运行时 UI 连接起来。本文档服务于后续实现 `VisualAssetRegistry` 与真实美术素材接入。
> **更新时间：** 2026-05-05

---

## 1. 设计目标

视觉资源系统要解决三个问题：

1. JSON 配置不能直接引用 Unity 对象。
2. 美术资源需要可替换，不能牵动核心玩法逻辑。
3. 缺失资源不能阻断 MVP 可玩流程。

因此程序侧采用：

```text
JSON Config -> VisualID -> VisualAssetRegistry -> Sprite / Prefab / AudioClip / VFX
```

---

## 2. 职责边界

### 2.1 JSON 配置
负责声明“这个实体需要什么表现”。

示例字段：

```json
{
  "ConfigID": "gear_tactical_blade",
  "Name": "战术长刀",
  "IconID": "item_gear_tactical_blade_icon",
  "WorldVisualID": "item_gear_tactical_blade_world",
  "UseVFXID": "vfx_weapon_slash_use",
  "UseSFXID": "sfx_weapon_slash_use"
}
```

JSON 不负责：
* Sprite 引用。
* Prefab 引用。
* Unity 导入设置。
* UI 尺寸布局。

### 2.2 VisualAssetRegistry
负责把稳定 ID 映射到 Unity 资源。

推荐资源类型：
* Sprite
* GameObject Prefab
* AudioClip
* Material
* AnimatorOverrideController

### 2.3 表现层
负责根据 VisualID 获取资源并显示。

示例：
* 背包物品 UI 根据 `IconID` 设置 `Image.sprite`。
* 战斗敌人卡片根据 `PortraitID` 设置头像。
* 战斗场景根据 `CombatVisualID` 实例化怪物 Prefab。
* 使用道具时根据 `UseVFXID` 和 `UseSFXID` 播放表现。

### 2.4 领域逻辑
领域逻辑不依赖 Sprite、Prefab、AudioClip。

示例：
* `Effect` 只负责回血、护盾、伤害等规则。
* `ItemUseService` 只负责使用校验和效果执行。
* `CombatNode` 只负责生成掉落和结算节点。

## 3. 推荐数据结构

### 3.1 配置实体新增字段

`ItemEntity` 推荐新增：

```csharp
public string IconID;
public string WorldVisualID;
public string UseVFXID;
public string UseSFXID;
public string RarityFrameID;
```

`MonsterEntity` 推荐新增：

```csharp
public string PortraitID;
public string CombatVisualID;
public string HitVFXID;
public string DeathVFXID;
```

节点配置或节点池条目推荐新增：

```csharp
public string NodeIconID;
public string NodeFrameID;
```

### 3.2 Registry Entry

推荐结构：

```csharp
[System.Serializable]
public class VisualAssetEntry {
    public string VisualID;
    public Sprite Sprite;
    public GameObject Prefab;
    public AudioClip AudioClip;
    public Material Material;
}
```

为了 MVP 简单，可以先使用一个统一 Entry。后续资源类型变多后，再拆成 `SpriteEntry`、`PrefabEntry`、`AudioEntry`。

### 3.3 Registry Asset

推荐使用 ScriptableObject：

```csharp
[CreateAssetMenu(menuName = "P3/Visual Asset Registry")]
public class VisualAssetRegistry : ScriptableObject {
    public List<VisualAssetEntry> Entries;
    public Sprite MissingSprite;
    public GameObject MissingPrefab;
}
```

原因：
* Unity 资源引用适合放在 ScriptableObject。
* JSON 继续保持纯数据。
* 美术替换资源时不需要改代码。

## 4. 运行时访问方式

### 4.1 服务入口

推荐提供一个轻量服务：

```csharp
public static class VisualAssetService {
    public static Sprite GetSprite(string visualID);
    public static GameObject GetPrefab(string visualID);
    public static AudioClip GetAudioClip(string visualID);
}
```

MVP 阶段可以由 `GameRoot` 或一个场景单例持有 Registry。后续再考虑 Addressables 或资源包。

### 4.2 缺失资源策略

任何资源缺失都不应直接让流程中断。

规则：
* `VisualID` 为空：返回默认占位图，并打一条 Warning。
* Registry 中找不到：返回默认占位图，并打一条 Warning。
* 资源类型不匹配：返回对应类型 fallback，并打一条 Error。

这样 AI 图未完成时，玩法仍然可测。

## 6. Prefab 使用规范

Prefab 应负责表现结构，而不是玩法事实。

可以放在 Prefab 中：
* SpriteRenderer 或 Image。
* Animator。
* 粒子挂点。
* 受击点。
* 音效源。
* UI 子结构。

不应放在 Prefab 中：
* 物品价格。
* 背包占用形状。
* 怪物 HP、伤害、掉落池。
* 道具效果数值。
* 地牢节点逻辑。

---

## 7. 配置校验

后续 `ValidateAllConfigs()` 应增加视觉资源校验：

* Item 的 `IconID` 是否存在。
* Monster 的 `PortraitID` 是否存在。
* Node 的 `NodeIconID` 是否存在。
* Registry 中是否有重复 VisualID。
* Registry 中是否存在未使用资源。
* 资源类型是否匹配字段用途。

校验结果建议输出到：

```text
UnityClient/Logs/VisualAssetValidationReport.json
```

## 8. 开发里程碑

### 8.1 第一阶段：最小可用接入

目标：
* 新增 `VisualAssetRegistry`。
* 新增 `VisualAssetService`。
* `ItemEntity` 支持 `IconID`。
* `DraggableItemUI` 显示真实图标。
* 缺失资源显示占位图。

验收：
* 背包、战利品、工坊中物品图标一致。
* 没有素材时仍能跑完整 MVP 闭环。

### 8.2 第二阶段：怪物和节点表现

目标：
* `MonsterEntity` 支持 `PortraitID`。
* 战斗敌人列表显示头像。
* 节点按钮显示图标。

验收：
* 玩家能通过图形区分怪物和节点类型。
* 目标选择点击区域不被图片遮挡。

### 8.3 第三阶段：Prefab 与特效

目标：
* 怪物战斗 Prefab 接入。
* 道具使用 VFX/SFX 接入。
* 武器攻击表现走 `VisualQueue`。

验收：
* 表现延迟不影响领域事件推进。
* 自动化测试仍可在 headless 模式下绕过表现。

---

## 9. 自动化测试建议

新增测试方向：

* Registry 重复 ID 检查。
* JSON 中视觉 ID 缺失检查。
* `VisualAssetService` fallback 检查。
* `DraggableItemUI.SetupData` 在无图标时不报错。
* Headless 模式下视觉资源缺失不影响战斗、拾取和卖出。

---

## 10. 当前项目接入顺序建议

第一步：
* 物品图标 Registry。
* 背包、战利品、工坊统一显示物品图标。

第二步：
* 怪物头像 Registry。
* 战斗敌人占位卡替换为头像卡。

第三步：
* 节点图标 Registry。
* 地图按钮图形化。

第四步：
* 背景图和战斗 Prefab。
* VFX/SFX。

这个顺序能最快把 AI 美术产物放进当前 MVP 闭环，同时不打断已有规则和回归测试。
