# 客户端核心架构规范 (Client Architecture Guidelines)

> **定位说明：** 本文档为客户端程序的最高架构指导准则。阐述了如何通过严格的职责分离、事件驱动模型、表现层解耦与反射工厂等现代软件工程模式，构建一个高内聚、低耦合、高度可测试且具备工业级稳定性的“数据驱动型”游戏工程。

---

## 1. 宏观架构范式 (Macro-Architectural Paradigms)

本项目的核心架构基调确立为：**“将 Unity 引擎降维为纯粹的渲染视图（View）与输入设备（Controller），核心业务逻辑（Model）实现完全独立”**。

### 1.1 极致的关注点分离 (Strict MVC & Headless Backend)
*   **纯 C# 领域模型 (POCO Domain Model)**：贯彻“无头黑盒 (Headless Sandbox)”设计原则。游戏的核心领域逻辑（包含但不限于：战斗演演、网格碰撞矩阵解算、背包拓扑结构管理、核心状态机流转）必须采用纯 C# POCO 实现。**在领域层绝对禁止继承或引用 `UnityEngine.MonoBehaviour` 及任何具体的表现层 API**。
*   **数据驱动架构 (Data-Driven Architecture, DDA)**：万物皆配置。游戏内的所有静态实体元数据（人偶、怪物、物品、地图节点树）均由外部 JSON 配置文件强类型定义。引擎在初始化阶段完成反序列化注入，实现逻辑实现与数值调优的彻底物理隔离。
*   **状态与表现的物理隔离 (State & View Isolation)**：前端 UI 组件严格遵循无状态（Stateless）原则，绝不允许直接持底层的写权限（如 `CurrentHP -= 10`）。UI 仅作为底层数据快照的镜像呈现，所有的状态变更必须通过封装好的底层 API 接口发起请求（Request）。

### 1.2 全局事件总线模型 (Event-Driven Architecture, EDA)
*   **发布-订阅机制 (Pub-Sub Pattern)**：模块之间的横向通信，以及底层逻辑（Backend）与表现层（View）的纵向通信，强制使用强类型的 `EventBus` 模式。
*   **单向数据流 (Unidirectional Data Flow)**：底层逻辑完成瞬时结算后，仅负责向上层系统与订阅者“广播（Publish）”状态变更的事件快照。前端 UI 仅负责“监听（Subscribe）”并根据快照驱动画面渲染，从根本上杜绝“面条式代码 (Spaghetti Code)”与紧耦合陷阱。

### 1.3 视觉表现队列 (Visual Queue / Command Pattern)
*   **时序解耦与排队渲染 (Temporal Decoupling)**：面对高密度的回合制连环结算（如多重 Buff 触发、连锁反应），确立**“底层瞬间解算，前端排队渲染”**的核心准则。
*   **命令模式封装 (Command Encapsulation)**：后端在一帧（数毫秒）内完成全量逻辑推演后，将本应瞬间生效的视觉反馈（特效、飘字、血条位移）封装为实现了 `IEnumerator` 的 `IVisualCommand` 指令实体。
*   **异步执行机 (Asynchronous Runner)**：由专门的 `VisualQueueRunner` 协程机负责按序弹出队列指令，并注入时间延迟（DeltaTime）进行播放。此机制完美解决了底层毫秒级计算与前端动画耗时之间的帧同步冲突，杜绝了逻辑剧透现象。

### 1.4 反射机制与抽象工厂 (Reflection & Abstract Factory)
*   **开闭原则践行 (Open-Closed Principle, OCP)**：在处理高频迭代的策略模块（如技能效果 `EffectData` 的解析、地牢节点 `NodeBase` 的生成）时，严禁使用硬编码的 `if-else` 或 `switch-case` 庞大分支。
*   **动态装配 (Dynamic Discovery)**：采用 C# `System.Reflection` 机制。框架在启动时动态扫描当前 AppDomain 内的特定基类或接口派生类型，实现工厂模式（如 `EffectFactory`, `NodeFactory`）的自动化注册与多态实例化，大幅降低新增业务特性的接入成本。

## 2. 核心模块边界与分层约束 (Module Boundaries & Layering)

整个工程在代码结构上划分为以下严格的抽象层级：

### 2.1 引擎生命周期适配层 (Engine Adapter Layer)
*   **`GameRoot.cs`**：全游戏生命周期的唯一挂载入口。负责在 `Awake()` 阶段完成底层 `CoreBackend` 的单例引导与装配，并在 `Update()` 帧循环中将 Unity 的时间脉冲（Time Pulse）注入底层后端，驱动逻辑帧运转。
*   **`FileLogger.cs`**：系统级日志守护进程。接管 Unity Console 日志流，强制持久化至本地 `Logs/` 目录，为无头测试（Headless Testing）和崩溃追溯提供纯净的黑盒体检报告。

### 2.2 核心业务总控中心 (Core Backend Context)
*   **`CoreBackend.cs`**：作为领域逻辑的最高级上下文（Context）。它统筹所有基础系统的引导（如 JSON 反序列化），并作为各业务子系统（`PlayerProfile` 资产库、`CombatSystem` 战斗状态机、`DungeonManager` 副本图论树、`WorkshopSystem` 养成建造）的统一访问代理（Facade）。

### 2.3 贫血数据模型与系统计算器 (Anemic Domain Model & Solvers)
*   **数据载体 (`Entity`)**：如 `ItemEntity`, `DollEntity`。采用贫血模型（Anemic Model）设计，仅作为序列化数据的容器，不包含任何复杂的运算闭环。
*   **解算器 (`Solver` / `System`)**：业务运算全部剥离至无状态的静态解算器或管理器中（如 `GridSolver` 负责拓扑连结计算，`CombatSystem` 负责伤害减免公式），提升系统的横向扩展能力。

### 2.4 领域事件总线矩阵 (Domain Event Bus Network)
按业务边界拆分的细粒度事件中心：
*   **`GameEventBus`**：处理全局通用的生命周期与核心数值（HP/SAN/AP）变更事件。
*   **`CombatEventBus`**：处理高度局域化、高频的战斗轴内阶段事件（如 回合初、结算末 钩子触发）。
*   **`DungeonEventBus`**：处理副本状态机演进事件（如 节点跳转、深渊撤离、最终结算）。

### 2.5 视图渲染层 (View Presentation Layer)
*   **纯粹的响应式组件**：如 `DraggableItemUI`, `HUDController`。它们挂载于 UGUI 节点之上，其内部逻辑极尽削薄。仅在接收到 EventBus 或 VisualQueue 投递的命令时，执行动画插值或文本替换，充当底层逻辑状态最真实的只读监视器。
