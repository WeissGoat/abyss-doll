# 客户端核心架构规范 (Client Architecture Guidelines)

> **定位说明：** 本文档为客户端程序的最高架构指导准则。阐述了如何通过数据隔离、事件解耦与反射工厂等设计模式，构建一个高内聚、易测试、可复现的“数据驱动型”游戏工程。

---

## 1. 宏观架构模式：数据驱动 MVC 与纯 C# 黑盒

本项目的核心架构基调为：**“将 Unity 视为一个纯粹的渲染引擎与输入捕获器”**。所有的核心游戏规则均运行在一个独立于 Unity 场景树（Hierarchy）的纯 C# 环境中。

*   **无头逻辑黑盒 (Headless Backend)**：
    *   背包校验、战斗结算、回合状态流转等所有业务逻辑，**绝不继承 `MonoBehaviour`**。
    *   它们作为纯粹的 C# 类（POCO）存活在内存中。这种设计使得业务逻辑极易进行单元测试，并且为未来的断点存档、战斗回放打下基础。
*   **状态与表现的绝对隔离 (State & View Decoupling)**：
    *   **UI 的边界**：前端（View）脚本仅负责两件事：捕获玩家输入，以及更新屏幕像素。
    *   **单向数据流**：前端绝不允许直接修改底层数据（如 `CurrentHP -= 10`）。前端只能发起请求（Request），底层计算完毕后通过 **事件总线 (EventBus)** 广播数据变化的快照，前端监听到事件后再执行相应的画面刷新。

## 2. 核心模块的分层设计

整个工程在代码层面上划分为以下明确的层级：

### 2.1 引擎生命周期入口 (Bootstrapper)
*   **`GameRoot.cs`**：全游戏唯一的入口 `MonoBehaviour`，挂载于常驻场景中。它负责在 `Awake()` 阶段实例化底层的 `CoreBackend`，并在 `Update()` 中将其转化为后端的 `Tick(deltaTime)`，驱动底层时间流动。
*   **`FileLogger.cs`**：系统级组件，拦截并接管 Unity 的所有 Console 输出，强制写入本地 `Logs/` 目录，以支持在无法看到 UI 时的“黑盒调试”。

### 2.2 业务逻辑总控 (Core Backend)
*   **`CoreBackend.cs`**：纯 C# 的全局访问点（采用依赖注入或服务定位器模式）。它不仅统筹初始化所有数据配置，还作为各个子系统（如 `PlayerProfile`、`CombatSystem`、`DungeonManager`、`WorkshopSystem`）的统一挂载上下文。

### 2.3 数据实体层 (Data Entities)
*   **`ItemEntity`, `DollEntity`, `MonsterEntity`**：这些是纯数据的结构体载体。不包含任何游戏机制的运算方法（运算由 `System` 和 `Solver` 处理），完全作为数据仓库存在。这种“贫血模型”配合系统组件化设计，极大地提升了系统的拓展性。

### 2.4 全局事件总线 (Event Bus Network)
按业务领域拆分的静态事件分发中心，用于切断逻辑层与表现层的硬引用。
*   **`GameEventBus`**：处理全局通用的 UI 更新事件（如 `OnAPChanged`, `OnHPChanged`, `OnItemPlaced`）。
*   **`CombatEventBus`**：处理高度局域化的战斗轴事件（如 `OnTurnStart`, `OnTurnEnd`）。
*   **`DungeonEventBus`**：处理副本推进与场景切换事件。

### 2.5 表现层 (View & UI)
*   **`DraggableItemUI`, `GridSlotUI`, `HUDController` 等**：挂载在 Canvas 上的组件。所有的渲染层脚本都遵循无状态化原则，其自身的呈现结果永远是底层数据的真实镜像。
