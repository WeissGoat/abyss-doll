# AI 素材生成与筛选流程

> **定位：** 约定 AI 图片生成、筛选、命名、验收和进入 Unity 工程的流程，避免原始素材污染正式工程。
> **更新时间：** 2026-05-05

---

## 1. 流程目标

AI 素材工作流的目标不是“一次生成最终资产”，而是快速获得可筛选的方向稿，再把少量稳定结果转成工程可引用资源。

流程分为四层：

```text
生成 -> 筛选 -> 整理 -> 接入
```

每一层都有明确产物：

* 生成：大量候选图。
* 筛选：少量可用方向。
* 整理：命名、裁切、透明背景、尺寸统一。
* 接入：进入 `Approved` 并登记 `VisualID`。

---

## 2. 目录流转

```text
UnityClient/Assets/Art/_IncomingAI/
  2026-05-05_items_batch01/
    raw/
    selected/
    notes.md

UnityClient/Assets/Art/Approved/
  Items/Icons/
  Monsters/Portraits/
  Nodes/Icons/
```

### 2.1 `_IncomingAI/<batch>/raw`
存放 AI 原始输出。允许保留模型默认文件名，允许混乱。

### 2.2 `_IncomingAI/<batch>/selected`
存放本批次初筛通过的候选图。可以继续保留临时命名，但需要在 `notes.md` 中写清楚用途。

### 2.3 `Approved`
存放准备接入游戏的版本。进入这里后必须遵守命名、尺寸和用途规范。

---

## 3. 批次命名

推荐格式：

```text
YYYY-MM-DD_<domain>_<topic>_batchNN
```

示例：

```text
2026-05-05_items_weapons_batch01
2026-05-05_items_consumables_batch01
2026-05-05_monsters_layer1_batch01
2026-05-05_nodes_mvp_batch01
```

---

每个批次建议包含一个 `notes.md`：

```markdown

## 目标
生成 MVP 阶段武器图标，用于背包、战利品拾取和工坊卖出列表。

## 生成提示词
...

## 筛选结果
- selected_01: 战术长刀方向，适合 `gear_tactical_blade`

```

## 5. 生成规格

### 5.1 物品图标

推荐提示词方向：

```text
single game item icon, isolated object, transparent background, readable silhouette, centered composition, dark fantasy sci-fi scrap aesthetic, no text, no watermark
```

规格：
* 目标尺寸：512x512
* 背景：透明
* 重点：轮廓清楚，小尺寸可读

验收标准：
* 物品主体不贴边。
* 64x64 缩略图仍能看清类别。
* 不包含文字、水印、签名。
* 不与已有物品图标过于相似。

### 5.2 怪物头像

推荐提示词方向：

```text
monster portrait, front facing or three quarter view, readable silhouette, hostile scavenger creature, dark abyss exploration game, no text, no watermark
```

规格：
* 目标尺寸：1024x1024 或 1024x1536
* 背景：透明或统一暗色背景
* 重点：面部和轮廓有辨识度

验收标准：
* 能在战斗敌人卡片中清楚识别。
* 层级怪物之间有明显差异。
* 不要过度复杂导致缩小后糊成一团。

### 5.3 节点图标

推荐提示词方向：

```text
minimal game map node icon, clear symbol, dark abyss dungeon, high contrast, no text, transparent background
```

规格：
* 目标尺寸：512x512
* 背景：透明
* 类型：Combat、SafeRoom、Boss、Event、Exit

验收标准：
* 同一节点类型风格统一。
* 不同节点类型能一眼区分。
* 图标能适配地图按钮。

## 6. 筛选标准

每张候选图按以下维度评估：

* 玩法识别：玩家能否立刻知道它是什么。
* 风格一致：是否符合“深渊、搜打撤、废土义体、魔偶工坊”的方向。
* 小尺寸可读：缩小到 UI 实际尺寸后是否仍然清楚。
* 工程可用：是否方便裁切、去底、压缩和复用。
* 内容安全：是否有水印、文字、侵权特征或不适合上线的元素。

不通过的典型原因：

* 构图太满，UI 中容易被裁。
* 细节过碎，小图不可读。
* 有随机文字、签名、水印。
* 与已有素材风格冲突太大。
* 图像内容表达了错误玩法，例如治疗道具看起来像武器。

## 7. 整理标准

通过筛选后，进入 `Approved` 前需要完成：

1. 裁切到目标尺寸。
2. 清理背景或统一底色。
3. 按命名规范重命名。
4. 确认用途后缀。
5. 记录对应 `ConfigID` 或计划绑定的 `VisualID`。

示例：

```text
raw/ChatGPT Image 2026-05-05 001.png
selected/repair_kit_candidate_01.png
Approved/Items/Icons/item_con_repair_kit_icon.png
```

---

## 8. VisualID 命名

VisualID 推荐与文件名保持接近，但不包含扩展名：

```text
item_con_repair_kit_icon
item_gear_tactical_blade_icon
monster_mob_scavenger_bug_portrait
node_safe_room_icon
bg_workshop_day
```

JSON 配置中只写 VisualID：

```json
{
  "ConfigID": "con_repair_kit",
  "Name": "便携修复剂",
  "IconID": "item_con_repair_kit_icon"
}
```

---

## 9. 接入验收

进入游戏前需要完成以下检查：

* Unity 中资源类型设置正确。
* Registry 中存在对应 VisualID。
* JSON 中引用的 VisualID 不为空。
* 运行时找不到资源时有占位图。
* 背包、战利品、工坊列表中显示正常。
* 图片没有挡住拖拽、点击和敌人选择的射线。

---

## 10. 版本管理建议

MVP 阶段建议：
* `_IncomingAI` 可只提交精选批次，海量原始废稿不进入 Git。
* `Approved` 中的资源应提交。
* `Runtime/Registries` 中的注册表应提交。
* 大图、音频、视频后续考虑 Git LFS。

在没有 Git LFS 前，避免一次提交大量未压缩大图。
