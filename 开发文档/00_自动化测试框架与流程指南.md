# 幽灵探针：全自动无头测试框架指南 (Ghost Daemon Auto-Test Framework)

> **文档定位**：本文档详细记录了 Project P3 中专门为“AI 驱动开发 (Auto-Agentic Workflow)”量身定制的底层自动化测试框架。该框架彻底打破了 Unity 必须依赖界面交互和手动点击 Play 的测试壁垒，实现了代码编写、编译、执行、验证的零人工干预闭环。

---

## 一、 工作流与工程化思维 (Workflow & Engineering)

### 1. 自动化优先与 TDD (Auto-Agentic Workflow)
*   **先验灵魂，再塑肉体**：在没有 UI 甚至是场景的情况下，必须先编写纯底层的单元测试 / 冒烟测试（Smoke Test）。
*   **幽灵探针 (Ghost Daemon)**：极度推崇自动化闭环。通过后台 `FileSystemWatcher` 监听外部指令并触发 `AssetDatabase.Refresh()`，实现“AI 写码 -> 自动编译 -> 反射跑测试 -> 输出 JSON 报告”的零人工干预测试流。只要底层逻辑（测试用例）没跑绿，绝不推进表现层。

### 2. 编辑器工具化 (Editor Automation)
*   **代码生成 UI**：极度反感在 Unity Inspector 中手动拖拽、连线、对齐 UI 预制体。
*   **幂等性构建**：倾向于编写极其硬核的 Editor 扩展（如 `MVPEditorSetup.cs`），一键 `DestroyIfExists` 并通过纯 C# 代码动态生成整个界面的层级树 (Hierarchy)、挂载组件和绑定引用。做到随时可推翻、随时可重建。

### 3. 务实的技术选型 (Pragmatic Tech Choices)
*   **实用主义至上**：不盲目追求新技术。例如在发现 UI Toolkit 在处理重度网格物理拖拽和射线穿透（Raycast）存在深层缺陷时，能够立刻果断止损，全局回退并统一使用经过验证的传统 `UGUI`，避免无谓的内耗。

## 二、 框架概述 (Overview)

在传统的 Unity 开发中，测试业务逻辑往往需要“编写代码 -> 挂载到 GameObject -> 点击 Play -> 观看动画 -> 查阅 Console”。这种流程极度消耗时间，且容易将底层数值 Bug 与前端表现 Bug 混淆。

为此，我们开发了 **“幽灵探针 (AutoTestDaemon)”**。它是一个常驻在 Unity 编辑器后台的无头 (Headless) 测试框架。它的核心使命是：**在没有任何场景、没有任何 UI、时间静止的情况下，以毫秒级的速度瞬间推演完极其复杂的 C# 后端系统逻辑，并输出结构化的体检报告。**

---

## 三、 核心架构与工作原理 (Architecture & Mechanics)

该框架的运转基于一条极其精妙的五步自动闭环：

1.  **静默监听 (File Watcher)**：
    驻留在 `Scripts/Editor/` 的 `AutoTestDaemon.cs` 利用 C# 的 `FileSystemWatcher`，死死盯住外部文件 `Logs/.test_trigger`。
2.  **外部触发 (CLI Trigger)**：
    AI 或开发者在外部命令行（PowerShell）中，将需要测试的类名与方法名（例如：`BackendSmokeTest.Run`）写入该触发文件。
3.  **强制重载 (Auto Compile)**：
    探针一旦监听到文件变更，立刻在主线程调用 `AssetDatabase.Refresh()`。这会强制 Unity 编译刚刚由 AI 写入的最新业务代码和测试脚本。
4.  **反射执行与拦截 (Reflection & Intercept)**：
    利用 `[DidReloadScripts]` 钩子，编译完成后探针瞬间苏醒。它通过 `System.Reflection` 动态定位到测试方法并执行。在此期间，探针会接管 `Application.logMessageReceived`，**全量拦截**所有的 `Debug.Log`、`LogWarning` 以及致命的 `Exception` 空指针。
5.  **生成体检报告 (Report Generation)**：
    测试跑完后，探针将截获的所有日志组合成一段 JSON 格式的数据，落地到 `Logs/TestReport.json`，并标记状态为 `PASSED` 或 `FAILED`。外部的 AI 读取该文件后，即可决定是继续开发，还是根据错误堆栈进行代码自愈。

## 四、 表现层降级机制 (The Headless Fallback)

测试框架最棘手的问题在于**“时间流逝”**。在实机游戏中，战斗表现队列 (`VisualQueue`) 会让一个火球飞 0.5 秒再扣血。如果在测试里也等 0.5 秒，测试效率将极其低下。

**降级原理：**
在 `VisualQueue.cs` 中，我们定义了全局静态变量 `public static bool IsHeadless = true;`。
*   当实机游戏启动（`GameFlowController` 或 `VisualQueueRunner` 运行）时，会将此值设为 `false`，开启排队缓冲。
*   但在探针跑测试时，没有任何 MonoBehaviour 会运行，该值保持 `true`。此时，`VisualQueue` 在接收到任何表现指令时，**会直接 `while (enumerator.MoveNext()) { }` 将内部的协程瞬间同步跑完**。这保证了即使底层逻辑中夹杂了 UI 表现代码，测试依然能在 1 毫秒内出结果。

## 五、 自动化测试流程 (SOP: Standard Operating Procedure)

当需要开发一个新系统（如：复杂的深渊掉落算法）时，标准流程如下：

### Step 1: 编写测试用例 (Test Case Writing)
创建一个纯静态类，编写不依赖场景的测试逻辑。
```csharp
public static class DropAlgorithmTest {
    public static void Run() {
        // 1. 初始化无头黑盒
        CoreBackend core = new CoreBackend();
        core.InitAllSystems();
        
        // 2. 强行注入静态上下文 (极其重要，替代 GameRoot 的作用)
        GameRoot.Core = core; 
        
        // 3. 构造极端测试环境
        var player = core.CurrentPlayer;
        player.Money = 0;
        
        // 4. 执行业务逻辑
        core.Dungeon.SettleDropLogic(); 
        
        // 5. 硬核断言 (Assert)
        if (player.StashInventory.Count > 0) {
            Debug.Log("Drop Logic PASSED.");
        } else {
            Debug.LogError("Drop Logic FAILED: Inventory is empty."); // LogError 会自动让报告变为 FAILED
        }
    }
}
```

### Step 2: 外部下达执行指令 (Triggering)
在 PowerShell 中执行：
```powershell
Set-Content -Path "UnityClient/Logs/.test_trigger" -Value "DropAlgorithmTest.Run"
```

### Step 3: 触发编译 (Compile)
*   （当前 Unity 机制限制）如果 Unity 处于后台失焦状态，开发者需要**切回 Unity 窗口点击一下鼠标**，让右下角的编译齿轮转动。

### Step 4: AI 静默抓取报告 (Fetch & Auto-Heal)
AI 在 PowerShell 中轮询读取报告：
```powershell
Get-Content "UnityClient/Logs/TestReport.json"
```
*   **若返回 FAILED**：AI 提取 JSON 中的 `[Exception]` 堆栈，自我反思，修改 C# 业务代码，并自动回到 Step 2。
*   **若返回 PASSED**：底层逻辑宣告竣工，可以通知前端同学动手绑定 UI。

## 六、 测试编写铁律 (Golden Rules for Tests)

1.  **绝对禁止 MonoBehaviour**：测试类必须是纯粹的 `public static class`，测试入口必须是 `public static void Run()`。
2.  **不要忘记上帝视角赋值**：实机中 `GameRoot` 负责把 `CoreBackend` 存入静态变量。在测试用例里，你必须手动写下 `GameRoot.Core = new CoreBackend();`，否则后续所有依靠 `GameRoot.Core` 寻址的单例全部会报空指针。
3.  **用 Error 宣判失败**：不需要引入复杂的 NUnit 框架。只要在测试中执行了 `Debug.LogError(...)` 或者抛出了任何未捕获的 `Exception`，探针就会自动将本次测试的 Status 宣判为 `FAILED`。

## 七、 测试编写避坑指南 (Common Gotchas)

### Common Mistake: 避免在断言中硬编码配置数据

**Symptom**: 修改了 JSON 配置表（如物品底价 `BaseValue`），业务逻辑依然正确，但自动化测试报错 `FAILED: Expected 1550, got 1400`。

**Cause**: 在测试用例的断言阶段，使用了手算或硬编码的数字（例如：`int expectedMoney = initialMoney + 250 + 800;`），而不是从创建的测试实体中动态获取属性。

**Fix**: 将硬编码的值替换为动态计算：
```csharp
// 不要这样做 (Bad)
// int expectedMoney = initialMoney + 250 + 800; 

// 应该这样做 (Good)
int expectedMoney = initialMoney + loot1.BaseValue + loot2.BaseValue;
```

**Prevention**: 在编写涉及配置表数据（如价格、伤害、容量）的测试断言时，**永远优先使用对象身上的属性 (`entity.Property`) 作为预期值的计算因子**。测试的目的是验证“逻辑计算过程是否正确”，而不是“配置表里的数字是不是固定的 250”。
