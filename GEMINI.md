# 项目概述

本目录包含了一款复杂RPG/模拟游戏的系统设计文档（GDD）和技术蓝图。该游戏的核心循环由三大支柱构成：**深渊探索**（战斗与搜刮）、**小镇经营**（经济与变现）以及**人偶养成**（好感与维修）。整个游戏体验受到严格的时间限制（存活天数），迫使玩家在资源管理和风险收益之间不断做出残酷的抉择。

## 目录结构与内容

本项目是一个游戏策划、数值、技术架构设计与Unity客户端工程混合的仓库。

*   **`设计文档/`:** 核心文件夹，包含详细且结构化的Markdown文件，记录了游戏各个系统的详尽设计（GDD）。
    *   `GDD_00_系统关联总图.md`: 最关键的文档，提供了俯瞰全局的“上帝视角”，阐述了12个子系统如何相互作用。
    *   `系统详案/`: 包含具体的设计文档（GDD_01至GDD_12），详细拆解了背包网格战斗、深渊地图遍历、物品生命周期、好感度双轨机制、势力订单等独立机制。
    *   `技术参考/程序架构与类设计规划.md`: 一份技术蓝图，全面拥抱基于组件的架构 (CBA)、工厂模式、阵营封装以及节点树预生成，用于指导代码实现。
*   **`版本规划/`:** 存放游戏开发的阶段性目标、推演方法与表现层需求。
    *   包含最小可玩版本 (MVP) 的核心闭环内容清单 (`00_最小MVP体验闭环内容清单.md`)、数值沙盘推演方法论 (`01_数值模型沙盘推演方法论.md`)。
    *   包含美术UI需求清单 (`01_MVP美术与UI需求清单.md`)、开发里程碑与节点规划 (`02_开发里程碑与节点规划.md`)、Unity编辑器实操步骤指南 (`02_Unity编辑器实操步骤指南.md`)。
*   **`开发文档/`:** 存放具体的程序实现指南与底层数据结构定义，是连接策划案与客户端代码的桥梁。
    *   包含核心程序架构与类设计草案 (`00_程序架构与类设计草案.md`)。
    *   包含核心数据与实体容器 (`01_核心数据与实体容器(CoreData).md`)、网格背包系统 (`02_网格背包与计算系统(GridSystem).md`)、深渊与战斗循环 (`03_深渊与战斗循环(DungeonCombat).md`)、工坊与养成逻辑 (`04_工坊与养成逻辑(WorkshopSystem).md`)、表现层架构与事件总线 (`05_表现层架构与事件总线(ViewAndEventBus).md`)等具体C#实现指南。
    *   `数据与实体定义/`: 将所有实体（全局/人偶/物品/深渊/经济等）拆分为核心属性与组件，指导JSON配置表的结构。
*   **`配置表(JSON)/`:** 存放游戏最小可玩版本（MVP）所需的全部 JSON 格式实体配置文件，严格遵循“一物一文件”的结构，是数据驱动架构的核心数据源。
*   **`数值模型设计/`:** 存放底层数值逻辑。
    *   确立了“1格(CV) = 100金币 = 10DPS”的基准价值与空间本位模型，以及经济循环、战斗伤害生存公式、深渊产出期望等核心计算体系。
*   **`草稿/`:** 包含早期的文本文件（`.txt`），记录了游戏各模块的头脑风暴和初步构思。
*   **`UnityClient/`:** Unity 官方工程目录。
    *   采用数据驱动的 MVC + EventBus（事件总线）架构，包含 `CoreBackend`（无头业务黑盒）与基于 `UI Toolkit` / `UGUI` 的表现层。
*   **`coreloop.png`**: 一张视觉图表，展示了游戏的核心玩法循环。

## 核心游戏概念

根据文档，游戏的核心冲突源于对极其有限的几大共享资源的零和博弈：
1.  **时间 (天数):** 玩家必须每天在危险但高收益的探索（推进死亡线），与安全但耗时的疗愈（浪费赚钱机会）之间二选一。
2.  **背包空间:** 极其严苛的网格博弈，武器、战利品和没有战力但提供情绪价值的纪念物在网格中竞争有限的位置。
3.  **理智 (SAN值):** 在深渊探索中会被持续消耗，需要花费资源和时间在地面恢复，是深渊探索可持续性的阀门。
4.  **经济系统:** 不断上涨的强制性房租和高昂的机体维护费用带来了持续的生存压力，迫使玩家不管赚多少钱都必须不断深入深渊。

## 用途

本目录作为游戏设计和技术规划的唯一事实来源（Single Source of Truth）。它供游戏策划、文案和程序员团队使用，以对齐系统机制的共识、理解修改特定系统可能带来的全局连锁反应，并根据提议的架构来组织未来的代码库开发。

## 开发环境与调试规范

1.  **开发环境:** Win10 + PowerShell。所有自动化构建脚本、终端操作及排查指令均需在此环境下运行。
2.  **日志先行 (Logging):** 在进行客户端 C# 或底层算法开发时，**必须多加日志 (Debug.Log/LogWarning/LogError)**。特别是核心的状态机切换、网格坐标变动、事件总线(EventBus)广播与监听处，必须有详尽的上下文日志，以支持“黑盒排查”。
3.  **日志查看方式:** 
    *   游戏运行的所有引擎控制台输出，均已被 `FileLogger.cs` 自动拦截并写入本地文件。
    *   日志文件固定存放在项目根目录的 **`UnityClient/Logs/`** 文件夹下，按时间戳命名（如 `GameLog_YYYYMMDD_HHMMSS.log`）。
    *   如果遇到 UI 卡死、无法拖拽或逻辑报错，AI 可以直接通过 PowerShell 命令（如 `Get-Content` 结合 `-Tail`）读取该目录下的最新日志，实现免界面的精准 Bug 定位。
